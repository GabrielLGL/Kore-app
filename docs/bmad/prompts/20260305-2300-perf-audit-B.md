# Rapport — Performance Audit — Groupe B (Startup / Lazy Loading) — 20260305-2300

## Objectif
Réduire le temps de démarrage de l'application en implémentant le lazy loading des écrans et en différant les imports lourds.

## Fichiers concernés
- `mobile/src/navigation/index.tsx` (imports L13-40, config stack)

## Contexte technique
- **22 écrans** sont importés de manière eagerly (statique) au top du fichier de navigation.
- Aucun `React.lazy()` ni `import()` dynamique n'est utilisé.
- Au démarrage, TOUS les écrans sont chargés en mémoire, même ceux rarement visités (Assistant, Stats*, Badges, ExerciseCatalog, etc.).
- Stack : React Native Expo 52, Fabric, React Navigation 7 (Native Stack).
- L'app a un AnimatedSplash (`components/AnimatedSplash.tsx`) qui affiche un splash pendant 1.65s.
- DB init : `mobile/src/model/index.ts`.

## Étapes

1. **Implémenter React.lazy() pour les écrans secondaires :**
   - Garder en import statique les écrans critiques du flow principal : `HomeScreen`, `OnboardingScreen`
   - Convertir tous les autres en lazy imports :
     ```tsx
     const ProgramsScreen = React.lazy(() => import('../screens/ProgramsScreen'))
     const SessionDetailScreen = React.lazy(() => import('../screens/SessionDetailScreen'))
     // etc.
     ```
   - **Note importante :** React Navigation Native Stack supporte les composants lazy. Wraper dans `React.Suspense` avec un fallback léger (ActivityIndicator centré sur fond `colors.background`).

2. **Créer un composant Suspense fallback réutilisable :**
   ```tsx
   // mobile/src/components/ScreenLoader.tsx
   const ScreenLoader = () => (
     <View style={{ flex: 1, justifyContent: 'center', alignItems: 'center', backgroundColor: colors.background }}>
       <ActivityIndicator color={colors.primary} />
     </View>
   )
   ```

3. **Wrapper chaque Screen.Screen component :**
   ```tsx
   <Stack.Screen name="Programs" component={ProgramsScreen} />
   // React.Suspense boundary au niveau du NavigationContainer ou de chaque screen
   ```
   - Option A : un `<Suspense>` global autour du `<Stack.Navigator>`
   - Option B : des `<Suspense>` par screen (plus granulaire)
   - **Recommandation : Option A** — plus simple, un seul fallback.

4. **Vérifier que les exports sont `export default` dans chaque écran** (requis pour lazy import). Scanner les fichiers.

## Contraintes
- Ne pas casser la navigation (types `RootStackParamList` inchangés)
- Ne pas modifier les écrans eux-mêmes, uniquement la navigation
- Le splash screen existant (AnimatedSplash) doit continuer à fonctionner
- Tester que le deep linking fonctionne toujours si applicable

## Critères de validation
- `npx tsc --noEmit` → zéro erreur
- `npm test` → zéro fail
- L'app démarre sans crash
- Navigation vers chaque écran fonctionne (pas de blank screen permanent)
- Temps de startup perceptiblement réduit (avant : tous les écrans chargés, après : seuls Home + Onboarding)

Aucune dépendance — peut être lancé en parallèle avec A et C.
